# SOC 告警运营：降噪分析 Tab 设计

## 背景

当前 `webui/src/pages/Soc/Alerts.tsx` 的告警运营页包含一条从原始告警到处置建议的运营漏斗，并在漏斗下方提供两个页签：

- 研判告警
- 深度调查

本设计新增一个位于“研判告警”左侧的页签：**降噪分析**。

该页签用于解释告警在进入研判前是如何被过滤、合并和压缩的，重点回答三类问题：

1. 今天原始告警减少了多少，分别由哪些降噪逻辑贡献。
2. 每类降噪逻辑命中了哪些条件、规则、黑白名单、扫描识别和重复聚合。
3. 被过滤或合并的告警是否可解释、可审计、可下钻。

本阶段只使用 mock 数据实现展示，不接真实接口。

## Workflow 依据

参考本机 workflow：

```text
~/.flocks/plugins/workflows/stream_alert_denoise/
```

该 workflow 文档名为 `stream_alert_dedup`，实际目录为 `stream_alert_denoise`，核心链路为：

```text
receive_alert
  -> normalize
  -> filter_logs
  -> dedup_and_write
```

核心统计字段：

| 字段 | 含义 | 页面使用方式 |
|---|---|---|
| `raw_count` | 原始输入告警数 | 顶部漏斗与降噪总览 |
| `normalized_count` | 归一化后告警数 | 处理链路健康度 |
| `after_filter_count` | 过滤后保留数 | 过滤阶段输出 |
| `filter_removed_count` | 过滤剔除数 | 降噪贡献 |
| `filter_process_type_counts` | 过滤分类计数 | 扫描、非 HTTP、方向分类 |
| `after_dedup_count` | 去重处理总数 | 去重前基数 |
| `unique_key_count` | 唯一 dedup key 数 | 去重后唯一告警 |
| `dedup_removed_count` | 重复告警减少数 | 重复告警贡献 |
| `dedup_ratio` | 批内压缩率 | 去重效率 |
| `lsh_total_clusters` | LSH 簇数量 | 去重状态与趋势 |
| `lsh_total_dedup_keys` | 已见 dedup key 数 | 历史去重状态 |

Workflow 目前过滤阶段主要通过 `_process_type` 分类保留非扫描 HTTP 告警，去重阶段通过 `strict_fields` 和 `lsh_fields` 生成 `dedup_key`。UI 设计在此基础上补充“条件、规则、黑白名单”等业务解释维度，作为 mock 展示数据。

## 信息架构

新增后的页签顺序：

```text
降噪分析 | 研判告警 | 深度调查
```

默认进入 `/soc/alerts` 时建议默认打开“研判告警”，保持当前体验不变。用户点击“降噪分析”后进入本设计页面。

降噪分析页由四层组成：

```text
1. 日期与范围选择
2. 降噪总览
3. 降噪贡献拆解
4. 降噪逻辑详细分析
```

## 页面布局

### 一、顶部日期与范围工具条

位置：降噪分析 tab 内容区顶部。

控件：

- 日期选择：默认“今天”，支持切换最近 7 天中的某一天。
- 数据源筛选：全部、TDP、SkyEye、WAF、NDR、EDR。
- 方向筛选：全部、入站、出站、横向。
- 资产范围：全部资产、互联网暴露面、办公网、生产网、核心业务区。
- 导出按钮：展示为 mock，不实现真实导出。

展示口径：

```text
2026-06-14 00:00 - 23:59
来源：全部告警源
处理链路：归一化 -> 过滤 -> 去重 -> 研判
```

### 二、降噪总览

总览区复用当前页面卡片风格，建议使用 5 个紧凑指标卡：

| 指标 | 示例值 | 说明 |
|---|---:|---|
| 原始告警 | 9,836 | 进入 workflow 的告警总数 |
| 归一化成功 | 9,812 | 字段映射成功的告警 |
| 过滤减少 | 6,944 | 扫描、非 HTTP、白名单、低价值规则等减少量 |
| 去重减少 | 1,869 | 批内与跨批次重复告警减少量 |
| 进入研判 | 1,023 | 降噪后需要研判的告警 |

卡片下方增加一条横向 Sankey 或分段条形图，用于呈现：

```text
原始告警 9836
  -> 归一化 9812
  -> 过滤后 2882
  -> 去重后 1023
```

视觉原则：

- 过滤减少用红色浅底，不表达风险，只表达减少量。
- 进入研判使用当前页面主色。
- 归一化失败单独以灰色小标识展示，避免抢主视觉。

### 三、降噪贡献拆解

在总览下方展示一组“贡献排行卡片”，每张卡片代表一种降噪逻辑。卡片可点击进入对应详细分析页。

建议 mock 分类：

| 降噪逻辑 | 减少量 | 占原始告警 | 说明 |
|---|---:|---:|---|
| 扫描告警 | 3,428 | 34.9% | 外部扫描器、搜索引擎、测绘引擎、内部扫描 |
| 重复告警 | 1,869 | 19.0% | 相同源目 IP 与相似 URI/body 的 LSH 聚合 |
| 条件过滤 | 1,124 | 11.4% | 非 HTTP、方向不在研判范围、字段缺失 |
| 规则过滤 | 782 | 8.0% | 低价值规则、噪声规则、长期误报规则 |
| 白名单 | 511 | 5.2% | 企业授权扫描器、监控探测、健康检查 |
| 黑名单命中 | 227 | 2.3% | 已确认黑灰产或情报命中，压缩为情报事件 |

卡片内容结构：

```text
标题 + 减少量
贡献占比 + 环比变化
Top 3 命中原因
查看分析
```

例如“扫描告警”卡片：

```text
扫描告警
减少 3,428 条
占原始告警 34.9%，较昨日 +6.2%

Top 原因：
- 外部通用扫描器探测登录口
- 搜索引擎/测绘引擎访问公开路径
- 内部安全工具周期扫描
```

### 四、详细分析区

贡献卡片下方展示详细分析页面。交互上可采用同页二级 tab 或左侧列表 + 右侧报告。

推荐结构：

```text
左侧：降噪逻辑列表
  - 扫描告警
  - 重复告警
  - 条件过滤
  - 规则过滤
  - 黑白名单

右侧：对应逻辑的分析报告
```

为避免页面过重，初版只在 mock 数据中完整设计“扫描告警”，其他逻辑提供可展示但较简洁的摘要表和 Top 列表。

## 交互状态

需要覆盖以下状态：

- 默认态：展示今天的降噪日报。
- 切换日期：所有卡片、排行、报告同步变化。
- 点击贡献卡片：右侧详细分析切换到对应逻辑。
- 空数据：显示“当日暂无该类降噪命中”，保留筛选器。
- 数据异常：例如归一化失败高于阈值时，在总览区显示黄色提示。
- 可审计入口：每个详细分析页提供“查看样本”按钮，mock 展示即可。

## 视觉和实现约束

- 延续当前 SOC 页面风格：白底、浅灰边框、红色主操作、紧凑表格。
- 不做营销式大卡片，优先展示运营人员能快速扫描的信息密度。
- 页签区不嵌套过多卡片，详细分析用左右分栏或报告式区块。
- 所有数字均使用 mock 数据，避免调用真实 workflow 输出。
- 先实现静态展示，再考虑接入真实 `stats` 和 JSONL 审计文件。

## 后续实现拆分

建议分三步实现：

1. 新增 tab 状态和 mock 数据，完成“降噪分析”总览与贡献拆解。
2. 完成扫描告警分析日报，包括来源、资产、路径、状态码、内外部扫描表。
3. 补齐重复告警、条件过滤、规则过滤、黑白名单的详细页，并为“查看样本”预留 drawer。
