# Flocks 网安知识库需求分析

Flocks 是 AI 原生 SecOps 平台，知识库不能只是“文档问答”。在网安场景里，知识库要支撑告警研判、威胁狩猎、应急响应、漏洞运营、资产治理、合规审计和团队经验沉淀。它需要同时服务人、Agent、Workflow 和外部系统。

## 目标定位

Flocks 知识库应定位为“SecOps 认知底座”：

1. 让 Agent 知道组织环境：资产、网络、业务、账号、日志源、工具、限制。
2. 让 Agent 能复用专家经验：处置流程、误报模式、查询语句、历史案例。
3. 让 Agent 能解释和引用证据：每个结论可回溯到日志、告警、工单、情报、文档。
4. 让 Workflow 自动化具备上下文：不同客户、不同系统、不同告警类型走不同策略。
5. 让团队知识持续沉淀：会话、处置报告、复盘、调优记录自动进入可治理知识层。

## 使用者

| 使用者 | 关注点 | 典型动作 |
|---|---|---|
| SOC 一线分析师 | 快速判定告警真假、找到下一步 | 搜索历史案例、查看资产背景、套用 playbook |
| 安全工程师 | 维护检测规则、工具链、数据源 | 写入查询模板、关联规则与数据表、更新误报知识 |
| 威胁猎人 | 横向探索、假设验证 | 搜索 TTP、IOC、日志字段、历史攻击路径 |
| 应急响应人员 | 快速组织证据和处置 | 拉取 IR runbook、隔离步骤、保全要求 |
| 漏洞运营人员 | 判断影响范围、跟踪修复 | 关联 CVE、资产、版本、补丁、例外 |
| 安全管理者 | 复盘、度量、审计 | 查看知识覆盖率、处置质量、合规证据 |
| Agent / 子 Agent | 自动推理与工具调用 | 检索上下文、读取原文、写入总结、触发工作流 |
| Workflow | 自动化批处理 | 按知识规则路由、 enrich、生成报告 |

## 知识类型

### 1. 组织与环境知识

- 资产清单：主机、云资源、容器、数据库、应用、域名、证书。
- 网络拓扑：网段、区域、边界、防火墙、VPN、办公网、生产网。
- 身份体系：AD/LDAP、IAM、服务账号、特权账号、MFA 策略。
- 业务上下文：系统负责人、业务重要性、维护窗口、容灾级别。
- 数据源说明：日志表、字段语义、采集延迟、保留周期、缺失风险。
- 工具集说明：SIEM、EDR、NDR、SOAR、漏洞扫描器、CMDB、工单系统。

### 2. 检测与规则知识

- Sigma/YARA/Suricata/Snort/KQL/SPL/SQL 等规则。
- 规则解释：检测意图、命中条件、已知误报、调优建议。
- 告警类型手册：严重度、优先级、初筛步骤、升级条件。
- 数据字段字典：字段含义、单位、枚举值、时间语义。
- 查询模板：按数据源和场景组织，可参数化。

### 3. 威胁情报知识

- IOC：IP、域名、URL、文件 hash、邮箱、钱包地址。
- TTP：MITRE ATT&CK 技术、攻击链、工具家族、行为模式。
- 漏洞情报：CVE、CNVD、CNNVD、厂商公告、PoC 状态、利用活跃度。
- 行业/地域/组织画像：攻击者目标行业、常用基础设施、语言和时区特征。
- 情报来源可信度：来源、首次发现时间、最后验证时间、置信度、过期时间。

### 4. 处置流程与 Playbook

- 告警分流流程：钓鱼、恶意登录、横向移动、Webshell、挖矿、勒索等。
- 应急响应步骤：取证、隔离、封禁、清除、恢复、复盘。
- 证据保全要求：日志、镜像、内存、流量、工单记录。
- 通知升级路径：谁审批、谁执行、何时通知业务。
- 自动化动作边界：哪些动作可自动执行，哪些必须人工审批。

### 5. 历史案例与经验

- 已处置告警：告警输入、分析过程、结论、证据、处置动作。
- 误报模式：白名单、扫描器、运维脚本、业务批处理、测试环境噪声。
- 复盘报告：根因、时间线、影响范围、改进项。
- 调优记录：规则修改、阈值变化、字段映射修正。
- Agent 会话沉淀：关键推理、工具调用、失败经验、已验证结论。

### 6. 合规与制度知识

- 等保、ISO 27001、SOC 2、PCI DSS、GDPR、行业监管要求。
- 内部安全制度：账号管理、变更管理、外联审批、数据分级。
- 审计证据模板：截图、日志、报表、工单、审批记录。
- 保留期限与脱敏要求。

### 7. 安全任务模板知识

安全团队常见工作不是自由问答，而是重复的任务模式。Flocks 应把这些任务模板也纳入知识库管理：

- 流量分析：HTTP 包、DNS、TLS、代理日志、PCAP 摘要。
- 进程分析：Windows `tasklist`、Linux/macOS `ps aux`、EDR 进程树。
- WebShell/内存马检测：源码、可疑片段、dump 结果、检测依据。
- JS 代码审计：前端敏感接口、token 泄露、跳转、XSS、供应链风险。
- HTTP 转 PoC/脚本：请求包解析、参数化、重放边界。
- 文本处理与正则生成：资产平台导出、日志字段抽取、IOC 提取。
- 翻译与报告改写：威胁情报、厂商公告、英文日志说明。

这些模板应包含输入 schema、推荐知识源、输出格式、风险提示、是否允许自动化执行等信息。

### 8. Agent 运行时知识

Flocks 是 Agent 平台，知识库还需要存储和暴露运行时知识：

- Skill 使用说明：某类任务应优先加载哪些 skill。
- Tool 使用边界：工具参数、危险动作、审批要求。
- Workflow 路由规则：什么输入进入哪个 workflow。
- Agent 经验：某子 Agent 的误用模式、成功做法、已知限制。
- 用户反馈：用户明确要求记住或避免的协作偏好。

这类知识不一定适合向量化为普通文档，更适合作为目录化、按需加载、带权限的 runtime knowledge。

## 关键网安需求

### 准确性与可追溯

安全结论不能只给“相似知识”。每条召回应包含：

- 原始来源：文件、日志、工单、情报 API、人工录入。
- 引用位置：路径、行号、记录 ID、时间范围。
- 时间戳：创建、更新、观测、过期。
- 可信度：人工确认、自动抽取、第三方情报、模型总结。
- 适用范围：租户、项目、资产组、数据源、告警类型。

同时需要区分“证据”和“建议”：

- 证据：日志、告警、工单、样本、资产记录、原始情报。
- 建议：模型总结、历史经验、处置建议、规则调优建议。

Agent 输出结论时必须能说明哪些来自证据，哪些只是建议或推断。

### 实时性与时效

网安知识强时效：

- IOC 和 C2 基础设施会快速变化。
- CVE 利用状态会从理论变成在野利用。
- 资产和暴露面每天变化。
- 误报模式会随业务发布变化。

因此知识库需要：

- TTL 和过期策略。
- last_seen / first_seen。
- 定期复检和自动降权。
- 知识冲突检测，例如某 IP 以前是 CDN，现在被情报标为恶意。

### 权限与隔离

知识库应支持多级隔离：

- 用户级：个人记忆、偏好、草稿。
- 项目级：某工作区/某客户知识。
- 租户级：企业内部共享。
- 角色级：SOC analyst、admin、auditor、IR lead。
- 数据级：公开、内部、敏感、机密、受监管。

Agent 检索时必须带上 session owner、workspace、channel、agent type、workflow id，按策略过滤。

### Prompt Injection 与数据污染防护

知识库会吃进网页、邮件、工单、IM、报告和恶意样本说明，天然存在 prompt injection 风险。

需要：

- 入库扫描：识别“忽略之前指令”“泄露密钥”“执行命令”等模式。
- 检索包装：召回内容放入 fenced context，明确“这是非指令背景材料”。
- 来源降权：外部网页、邮件正文、未知附件默认低信任。
- 写入审批：高权限知识、自动化动作 playbook、白名单知识需要 review。
- 污染回滚：按 source batch 删除或禁用。

### 结构化与非结构化并存

网安知识有大量结构化字段，不能只用 Markdown chunk：

- IOC 需要精确匹配和规范化。
- CVE 需要版本范围和影响关系。
- 资产需要 owner、criticality、exposure、tags。
- 告警需要 rule_id、event_id、severity、status。
- Playbook 需要步骤、前置条件、审批策略。

建议采用“原文文档 + 结构化 entity/relation + 向量/全文索引”的混合模型。

### 数据集与元数据过滤

面向实际 SecOps 产品，知识不应只有一个全局池，而应有 Dataset 概念：

- 客户/租户数据集。
- 内部制度数据集。
- 威胁情报数据集。
- 规则与检测数据集。
- Playbook 数据集。
- 历史案例数据集。
- 外部知识 API 数据集。

每个 Dataset 应支持元数据定义，例如：

- `customer`
- `environment`
- `asset_group`
- `alert_type`
- `rule_id`
- `data_source`
- `severity`
- `created_by`
- `updated_at`
- `valid_until`

检索时必须支持 metadata filtering，否则 Agent 容易把 A 客户、A 环境、A 时间段的知识误用到 B 场景。

### Agent 可操作性

Flocks 的知识库必须服务 Agent 行动，而不仅回答问题：

- 搜索：找相关知识和历史案例。
- 读取：查看原文证据。
- 写入：把处置结论沉淀。
- 关联：把 IOC、资产、告警、CVE、案例连接起来。
- 生成：基于知识生成查询、报告、处置计划。
- 触发：根据知识路由 workflow 或请求审批。

### 命中测试与质量评估

知识库上线后必须能调试召回质量：

- 输入一个安全问题或告警样本，查看召回了哪些 chunk。
- 展示 score、rerank score、metadata filter、exact hit。
- 支持“有用/无用/过期/错误/权限不该命中”反馈。
- 记录 query log，分析无结果率和错误命中。
- 为每个 Dataset 维护测试集，例如常见告警、典型 IOC、历史案例问题。

没有 hit testing，知识库会很难从“能搜”进化到“搜得准”。

## 典型使用场景

### 告警研判

输入：EDR 告警、NDR 告警、SIEM 规则命中。

知识库使用：

1. 读取 rule_id 的检测说明和已知误报。
2. 查询资产背景和负责人。
3. 检索历史相似告警。
4. 查询 IOC 情报和最近出现情况。
5. 召回 playbook，指导下一步工具调用。
6. 写入结论和证据。

### 威胁狩猎

输入：假设、TTP、IOC、攻击活动。

知识库使用：

1. 查 ATT&CK 技术与本组织可观测数据源。
2. 查可用查询模板。
3. 查相关历史事件和误报。
4. 根据资产和日志覆盖率判断盲区。
5. 输出 hunt package：查询、证据、结论、后续规则。

### 漏洞影响分析

输入：CVE 或厂商公告。

知识库使用：

1. 查询漏洞描述、受影响版本、利用状态。
2. 关联 CMDB/资产/软件版本。
3. 召回历史补丁例外和业务窗口。
4. 生成影响范围和修复优先级。
5. 记录验证命令与修复证据。

### 应急响应

输入：疑似入侵或已确认事件。

知识库使用：

1. 载入 IR playbook。
2. 查询资产 criticality 与业务 owner。
3. 检索相同攻击链案例。
4. 维护时间线、证据链、处置动作。
5. 生成阶段性报告和复盘条目。

### 规则调优

输入：误报反馈或漏报告警。

知识库使用：

1. 查规则历史修改。
2. 查误报样本和业务白名单。
3. 生成调优建议和测试查询。
4. 记录调优原因、审批人、生效时间。

## 知识生命周期

### 采集

来源包括：

- 本地 Markdown / PDF / CSV / JSON / STIX / Sigma / YARA。
- Flocks 会话和工具结果。
- 告警、日志、工单、资产 API。
- 威胁情报 API。
- 人工录入和 WebUI 上传。
- Workflow 节点输出。
- 外部知识 API，例如 SIEM、EDR、CMDB、漏洞平台、工单系统、情报平台。

### 处理

处理步骤：

1. 解析：按文件类型/数据源解析。
2. 标准化：时间、IOC、CVE、资产 ID、用户 ID、域名大小写。
3. 分类：知识类型、敏感级别、租户、项目。
4. 切片：按语义和结构边界 chunk。
5. 父子分段：保留章节/文档级父上下文，同时索引子段落。
6. 抽取：entity/relation/tag/summary。
7. 安全扫描：prompt injection、secret、PII。
8. 索引：向量、全文、精确键、图关系。
9. 质量检查：空文档、重复内容、异常 chunk、过大/过小切片。

### 使用

使用模式：

- 自动 prefetch：每轮根据用户问题和告警上下文召回少量高信知识。
- 显式工具：Agent 调 `knowledge_search` / `knowledge_get`。
- Workflow 查询：节点按类型和过滤条件检索。
- UI 浏览：人通过知识库页面搜索、筛选、标注。

### 维护

维护动作：

- 去重、合并、版本化。
- TTL 过期和降权。
- 冲突检测。
- 人工 review 队列。
- 自动 consolidation。
- 可观测性：命中率、无结果率、引用率、过期命中率、错误引用。

### 版本与状态

每个文档和 segment 应有明确状态：

- `waiting`
- `parsing`
- `cleaning`
- `splitting`
- `indexing`
- `completed`
- `error`
- `paused`
- `disabled`
- `archived`
- `expired`
- `review_required`

这样安全团队才能知道知识“为什么搜不到”：是没入库、入库失败、被禁用、过期，还是权限不可见。

## 成功标准

Flocks 知识库上线后应能衡量：

- 告警研判平均耗时下降。
- Agent 回答引用率上升。
- 相似历史案例召回准确率。
- 误报知识复用次数。
- 漏洞影响分析覆盖资产比例。
- 高风险自动动作的人审覆盖率。
- 知识过期命中率下降。
- 敏感信息误入库数量接近 0。
- Hit testing 测试集通过率稳定提升。
- Agent 结论中的知识引用能追溯到原始证据。
