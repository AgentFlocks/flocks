# Flocks 服务启动与访问指南

## 摘要
本文说明 Flocks 安装完成后的标准启动方式、默认访问地址、常用 CLI 命令，以及“服务已启动但页面打不开”这类高频问题的排查顺序。标准入口是 `flocks start`，默认 WebUI 地址为 `http://127.0.0.1:5173`，默认后端地址为 `http://127.0.0.1:8000`。

## 适用场景
- 安装完成后需要启动服务
- 不确定 Flocks 当前是否正常运行
- 页面打不开，或怀疑前后端没有同时启动

## 标准启动方式
README 中给出的标准 CLI 流程如下：

```bash
flocks start
flocks status
flocks logs
flocks restart
flocks stop
```

其中：
- `flocks start`：启动后端和 WebUI
- `flocks status`：查看当前运行状态
- `flocks logs`：查看启动日志和运行日志
- `flocks restart`：显式全量重启
- `flocks stop`：停止服务

## 默认访问地址
默认情况下：
- WebUI：`http://127.0.0.1:5173`
- API：`http://127.0.0.1:8000`

需要注意的是，默认监听地址是本机回环地址，因此只允许当前机器访问。

## 启动后的推荐检查步骤
### 第一步：确认进程状态

```bash
flocks status
```

### 第二步：查看日志

```bash
flocks logs
```

### 第三步：浏览器打开默认地址
访问：

```text
http://127.0.0.1:5173
```

### 第四步：确认页面不是空白或持续报错
如果页面能打开但交互异常，通常要继续检查默认模型配置、网络代理或 WebUI 请求地址。

## 常见问题
### 为什么本机可以访问，但其他机器打不开？
因为默认监听的是 `127.0.0.1`。如果要允许其他机器访问，需要调整启动参数，详见远程部署指南。

### 为什么执行 `flocks start` 后等待很久？
README 明确说明，`flocks start` 默认会先构建 WebUI，再启动服务。首次启动或前端依赖变化后，等待时间会更长。

### 为什么页面能打开，但功能点进去报错？
这通常说明 WebUI 已启动，但后端接口、默认模型或浏览器能力存在问题。先看日志，再确认页面是否已完成首次配置。

## 高频出现的问题
- 安装后不知道必须再执行一次 `flocks start`
- 页面能打开，但仍然请求错误的后端地址
- 用户在远程服务器或虚拟机中部署后，误以为默认地址也能被外部访问
- 升级后重启异常，实际需要 `flocks stop` 后重新启动

## 最常用的排查顺序
1. `flocks status`
2. `flocks logs`
3. 打开 `http://127.0.0.1:5173`
4. 检查页面是否完成默认模型配置
5. 如果是远程机器，检查监听地址是否仍为 `127.0.0.1`
