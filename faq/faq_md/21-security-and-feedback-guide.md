# Flocks 安全与反馈指南

## 摘要
本文整理 Flocks 在公网暴露、数据提交给模型、安全漏洞反馈和历史版本风险自查方面的注意事项。对于面向生产环境的部署，最重要的原则是尽量减少暴露面、区分本机访问和公网访问、并在发现产品安全问题时走明确的反馈渠道。

## 适用场景
- 想把 Flocks 部署到公网或对外提供服务
- 关心数据发送给大模型前的安全性
- 想反馈 Flocks 自身漏洞或安全问题
- 想确认自己是否受某个历史版本问题影响

## 公网暴露的核心风险
如果把 WebUI 或 API 直接绑定到 `0.0.0.0` 并对公网开放，会带来：
- 更大的攻击面
- 未授权访问风险
- 后端接口暴露风险

因此更推荐：
- 只对外开放 WebUI
- 后端 API 保持本机可访问
- 配合防火墙白名单、安全组或反向代理鉴权

## 数据发给模型前的安全关注点
群聊中有人关心“发给大模型之前做了哪些脱敏”。现有文档没有展开成完整安全白皮书，因此更适合在 FAQ 中给出原则性说明：
- 生产环境应优先确认模型部署位置和数据边界
- 如果有高敏感数据，优先使用内网模型或私有模型
- 对公网模型服务应审查自身数据出站策略

## 历史版本风险自查
群聊中曾出现针对特定时间窗口、特定版本的风险排查讨论。这类问题的处理原则应是：
- 优先查看官方公告
- 明确受影响时间范围和版本范围
- 按公告要求做文件、版本或运行状态自查

## 漏洞与安全问题如何反馈
群聊中的社区口径是：
- 发现安全漏洞可先联系官方群管理或产品团队

对于正式场景，建议统一通过：
- 官方 Issue
- 官方安全反馈渠道
- 团队指定联系人

## 推荐实践
- 默认只本机访问
- 远程访问优先只开放 WebUI
- 对高敏感数据优先用内网模型
- 跟进版本公告和升级说明
