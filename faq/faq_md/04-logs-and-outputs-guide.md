# Flocks 日志与结果查看指南

## 摘要
本文说明 Flocks 运行过程中常见日志入口、结果输出位置以及“提示已保存但找不到文件”的常见原因。排障时优先使用 `flocks logs`，需要进一步定位时再结合 `~/.flocks/logs/`、容器挂载目录和工作目录输出进行检查。

## 适用场景
- 页面报错但提示信息不完整
- 对话失败、模型调用失败、工具调用失败
- 系统提示报告已保存，但本地找不到对应文件

## 最先看的日志入口
命令行优先使用：

```bash
flocks logs
```

这是最适合第一时间确认启动、重启、接口报错和前后端运行状态的入口。

## 本地常见日志位置
根据群聊中的实际排障经验，后端日志通常可以在下面的位置查看：

```text
~/.flocks/logs/backend.log
```

如果是 WebUI、容器或其他组件问题，还应结合对应前端日志或容器日志一起排查。

## 如何区分不同类型的问题
### 启动类问题
看 `flocks logs`，关注服务是否真正启动、端口是否绑定成功、WebUI 是否构建成功。

### 模型类问题
重点看后端日志，结合模型测试页面确认是否是平台超载、API 地址错误、权限问题或响应格式异常。

### 工具类问题
检查工具调用日志、权限配置、外部服务地址和认证信息是否正确。

### 页面类问题
如果页面能打开但交互异常，除了后端日志，还应结合浏览器控制台和 WebUI 请求地址一起看。

## 为什么提示“报告已保存”却找不到文件
这类问题在 Docker 场景下尤其常见，通常有三类原因：

1. 容器内已经生成文件，但没有挂载到宿主机
2. 你查看的是错误的工作目录
3. 输出落在了 `~/.flocks`、挂载目录或会话工作区，而不是当前 shell 目录

## Docker 场景的额外注意事项
如果你使用 Docker，必须确认相关目录已经正确挂载。否则，容器里的输出不会自动出现在宿主机当前目录。

至少应检查：
- `~/.flocks` 是否正确挂载
- 你期望查看的 outputs 目录是否在宿主机可见
- 容器是否使用了自定义工作目录

## 推荐排障顺序
1. `flocks logs`
2. `~/.flocks/logs/backend.log`
3. 模型管理页的测试结果
4. Docker 挂载目录或工作目录
5. 浏览器控制台与网络请求
