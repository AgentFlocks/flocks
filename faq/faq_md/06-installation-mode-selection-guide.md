# Flocks 安装方式选择指南

## 摘要
本文帮助你在终端安装、源码安装和 Docker 安装之间做选择。总体原则是：需要完整交互能力时优先终端安装或源码安装；希望快速起服务或环境隔离时优先 Docker；Windows 环境下如果一键安装不稳定，社区中更常建议源码安装或 Docker。

## 适用场景
- 不知道该选哪种安装方式
- Windows、WSL、ARM、云服务器等环境下犹豫如何部署
- 想兼顾稳定性、浏览器能力和运维成本

## 三种主流方式对比
### 终端安装
优点：
- 官方推荐路径
- 最适合完整能力验证
- WebUI、后端、CLI 都更容易调试

适合：
- 个人本机使用
- 需要浏览器自动化与网页登录
- 需要快速排查模型和集成问题

### 源码安装
优点：
- 更容易理解安装过程
- 更适合安装失败后的定向修复
- 对手动升级、查看脚本逻辑更友好

适合：
- 需要自己掌控安装细节
- 一键安装失败后切换方案
- 计划长期维护、需要源码级排障

### Docker 安装
优点：
- 环境隔离
- 清理和重建更直接
- 对宿主机环境污染更小

限制：
- README 已明确说明 Docker 模式下 `agent-browser` 的 headed 模式不可用
- 需要自行处理端口映射、挂载目录和权限

适合：
- 快速部署服务
- 服务器环境标准化运维
- 不依赖本机交互式浏览器登录

## 按系统环境给建议
### Windows
群聊中的实际反馈较多，结论相对一致：
- 一键安装和一键升级都更容易出问题
- 更推荐源码安装或 Docker
- 必须注意管理员 PowerShell

### Linux / macOS
通常优先终端安装或源码安装，成功率更高，后续升级也更顺畅。

### WSL / WSL2
社区反馈显示，WSL 下安装和尤其是更新时容易受到 Windows 侧 Node.js 影响，因此不属于最稳定选择。

### ARM
群聊经验中，ARM 平台兼容性相对一般，更常建议 Docker。

## 如何快速做决定
- 需要浏览器自动化和网页登录：优先本机终端安装或源码安装
- 需要标准化部署与快速重建：优先 Docker
- 一键安装失败后想最稳妥继续：优先源码安装
- Windows 环境想少踩坑：优先 Docker 或源码安装
