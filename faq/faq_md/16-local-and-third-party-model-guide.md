# Flocks 本地模型与第三方模型接入指南

## 摘要
本文聚焦本地模型、自建模型服务和第三方兼容 OpenAI API 的模型接入方式。实践中最常见的正确路径是：先创建 `OpenAI Compatible` 供应商，再添加具体模型，再到模型清单中测试连接。很多 404、端口不生效、模型列表不显示的问题，都是接入方式不匹配导致的。

## 适用场景
- 想接入本地部署模型
- 想接入第三方 API 网关
- 想使用 Ollama、vLLM、Qwen 或其他兼容 OpenAI API 的服务

## 最常见的接入方式
### OpenAI Compatible
这是群聊中被反复提到的主路径，适用于：
- 自建兼容 OpenAI 的模型接口
- 第三方兼容网关
- 某些本地模型服务

推荐顺序：
1. 创建 `OpenAI Compatible` provider
2. 保存 provider
3. 再创建模型
4. 到模型清单里测试

## 支持与不支持要怎么理解
从社区经验看，“支持某个模型”通常并不代表：
- 它所有部署方式都完全兼容
- 它一定能自动拉取模型列表
- 它一定能在群聊、私聊、复杂任务里表现一致

更准确的理解是：
- 只要你的服务足够兼容 OpenAI API，接入成功概率就更高

## 高频问题一：`404 not found`
这通常意味着：
- Base URL 不对
- 路径不兼容
- 你以为它是 OpenAI Compatible，但实际接口并不完全兼容

## 高频问题二：端口看起来没生效
群聊中有人反馈，自定义 API 地址带端口后，测试似乎仍然在走默认端口。这类问题通常需要检查：
- Base URL 是否完整
- 中间网关是否做了转发
- 实际测试请求打到了哪里

## 高频问题三：本地模型私聊正常，但群聊或复杂任务报错
这说明“接上了”和“稳定可用”不是一回事。

可能原因包括：
- 模型能力不足
- 响应格式在复杂任务中不稳定
- 平台或会话上下文更长时更容易出错

## 群聊中高频出现的问题
- Qwen、本地模型、兼容服务到底怎么接
- 填了 Base URL 但模型列表不显示
- 保存后还是需要去模型清单单独测试
- 本地模型简单对话可以，复杂对话不稳定

## 推荐策略
- 优先选择兼容度高、社区验证多的 OpenAI-Compatible 服务
- 先追求“稳定可用”，再追求“完全本地化”
- 本地模型接入成功后，一定要做多轮测试，而不只是一次简单调用
