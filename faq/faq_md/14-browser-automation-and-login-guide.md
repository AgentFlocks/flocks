# Flocks 浏览器自动化与网页登录指南

## 摘要
本文说明 Flocks 在网页浏览、登录页面接入和浏览器自动化方面的适用边界。核心结论是：本机安装最适合交互式网页登录；Docker 模式下不适合 headed 浏览器场景；云服务器上更偏向无头浏览器和自动化读取，而不是本地式交互登录。

## 适用场景
- 想让 Flocks 自动打开网页并读取内容
- 需要在登录页手动输入账号密码
- 使用 Docker 后发现浏览器能力和预期不一致

## Flocks 浏览器能力的基本边界
README 已明确指出：
- Docker 模式下 `agent-browser` 的 headed 模式暂不可用

这意味着：
- 本机安装更适合“打开浏览器，我来手动登录”这类场景
- Docker 更适合后台服务型场景，不适合依赖交互式浏览器窗口的流程

## 推荐场景一：本机安装 + 网页登录
如果你的目标是：
- 打开某个安全产品网页
- 由人手动输入账号密码
- 登录后继续让 Flocks 读取页面数据

最推荐的方案是本机安装。

## 推荐场景二：云服务器或远程主机
在远程机器上部署时，更适合：
- 通过无头浏览器自动化获取内容
- 使用 API 接入
- 通过中间脚本或其他方式替代人工登录

如果你真正需要“弹出浏览器让我手动登录”，远程主机并不是最理想的环境。

## Docker 的已知限制
群聊中多次确认：
- Docker 不能像本机安装那样交互式拉起浏览器窗口并输入账号密码
- Docker 用户经常误以为自己也能复现演示环境中的浏览器登录流程

## 浏览器兼容性问题
群聊中还出现过：
- 某些机器上 Chrome 调不起来
- 换系统自带浏览器后正常

这说明浏览器能力除了依赖 Flocks 自身，也可能受到宿主机浏览器环境影响。

## 排障建议
1. 先确认自己是不是 Docker 部署
2. 再确认当前是否真的需要交互式网页登录
3. 如果需要，优先改为本机安装
4. 如果只是取数，优先考虑 API 或无头浏览器路径
