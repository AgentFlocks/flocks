# Flocks 安装失败排查指南

## 摘要
本文聚焦 Flocks 安装过程中最常见的失败场景，包括 Node.js 安装失败、`flocks` 命令不可用、WebUI 构建失败、浏览器依赖失败以及安装脚本执行一半中断。排查时建议按“依赖是否齐全、脚本是否真正执行完、CLI 是否可用、WebUI 是否构建成功”的顺序推进。

## 适用场景
- 一键安装失败
- 安装提示完成，但实际无法启动
- `flocks` 命令不存在
- WebUI 构建失败或安装卡死

## 标准排查顺序
1. 检查 `uv`、`Node.js`、`npm 22+` 是否具备
2. 确认安装脚本是否完整执行完
3. 检查 `flocks` 命令是否可用
4. 尝试执行 `flocks start`
5. 查看 `flocks logs`
6. 如果仍异常，改用源码安装重新覆盖

## 高频问题一：Node.js 或 npm 安装失败
README 已明确说明，如果自动安装 `npm` 失败，请手动安装 `npm 22+`。

群聊中这类问题很常见，表现包括：
- 一键安装卡在 Node.js 步骤
- WebUI 构建失败
- 更新时前端重新构建失败

建议做法：
- 手动安装符合要求的 Node.js 与 npm
- 重新执行安装脚本

## 高频问题二：安装提示成功，但 `flocks` 命令不可用
这通常说明安装流程中有关键步骤失败，尤其是后端或 CLI 安装没有真正完成。

群聊中出现过的社区修复思路包括：

```bash
uv tool install -e . --force
```

但更稳妥的做法仍是：
- 进入源码目录
- 重新执行 `./scripts/install.sh` 或 `install.ps1`

## 高频问题三：WebUI 构建失败
常见原因：
- Node.js / npm 不符合要求
- 前端依赖没有正确安装
- 网络环境导致依赖下载中断

建议：
- 先解决 npm 版本和网络问题
- 再重新执行安装脚本

## 高频问题四：浏览器依赖失败
这不是一个小问题。群聊中的实际案例表明，浏览器依赖失败有时会导致后端、CLI 或整条安装链路不完整。

如果出现这种情况，建议：
- 不要直接假设“其他功能没事”
- 重新执行安装并观察报错位置
- 必要时切换为源码安装

## 高频问题五：安装一半卡住或长时间无响应
优先区分两种情况：
- 正在下载或构建，仍有真实进展
- 已经卡死，无新输出

如果长时间卡在同一步且没有新输出，优先考虑：
- 网络环境问题
- npm / Node.js 依赖问题
- 平台权限问题


## 什么时候建议直接重新安装
如果同时满足以下情况，通常比继续局部修补更省时间：
- 一键安装已经多次失败
- `flocks` 命令不可用
- WebUI 构建失败
- 你当前可以接受切换为源码安装
